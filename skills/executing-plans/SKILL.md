# 계획 실행

## 개요

계획(plan)을 배치 단위로 실행하고 각 체크포인트에서 검증합니다.

## 5단계 실행 프로세스

### 1단계: 계획 로드 및 검토

```
☐ 계획 문서 읽기
☐ 비판적 검토 (누락된 단계, 불명확한 부분)
☐ 필요시 계획 수정 요청
```

### 2단계: 배치 실행

```
기본 배치 크기: 3개 작업

실행 규칙:
☐ 작업 순서 준수 (의존성 고려)
☐ 각 작업 완료 후 즉시 검증
☐ 실패 시 즉시 중단 및 보고
```

### 3단계: 체크포인트 보고

각 배치 완료 후 보고:

```markdown
## 배치 N 완료

### 구현 결과
- [x] 작업 1: 완료 (검증: 테스트 통과)
- [x] 작업 2: 완료 (검증: 빌드 성공)
- [ ] 작업 3: 실패 (원인: ...)

### 다음 배치
- 작업 4, 5, 6 예정

### 이슈/질문
- (있으면 기재)
```

### 4단계: 피드백 반영

```
☐ 피드백 검토
☐ 필요시 계획 조정
☐ 다음 배치 진행 또는 수정
```

### 5단계: 완료 및 정리

```
☐ 모든 작업 완료 확인
☐ 전체 테스트 실행
☐ 최종 검증 (빌드, 린트)
☐ 결과 요약 보고
```

## 배치 실행 규칙

### 기본 원칙

| 규칙 | 설명 |
|------|------|
| 배치 크기 | 기본 3개, 복잡도에 따라 조정 |
| 검증 필수 | 각 작업 후 테스트/빌드 확인 |
| 실패 시 중단 | 다음 작업 진행 전 문제 해결 |
| 보고 주기 | 배치마다 진행 상황 공유 |

### 작업 우선순위

```
1. 의존성 없는 작업 먼저
2. 핵심 기능 → 부가 기능
3. 테스트 → 구현 (TDD 적용 시)
```

## 실행 중 체크리스트

```
각 작업 시작 전:
☐ 이전 작업 완료/검증됨?
☐ 필요한 컨텍스트 있음?
☐ 예상 결과 명확?

각 작업 완료 후:
☐ 테스트 통과?
☐ 빌드 성공?
☐ 예상 결과와 일치?

배치 완료 후:
☐ 진행 상황 보고?
☐ 이슈/질문 공유?
☐ 다음 배치 준비?
```

## 실패 처리

```
작업 실패 시:
1. 즉시 중단
2. 실패 원인 분석
3. systematic-debugging 적용
4. 해결 후 재실행
5. 필요시 계획 수정
```
