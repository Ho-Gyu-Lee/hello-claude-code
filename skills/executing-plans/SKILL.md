# 계획 실행

## 6단계 실행 프로세스

### 1단계: 계획 로드 및 검토

```
[ ] 계획 문서 읽기
[ ] 비판적 검토 (누락된 단계, 불명확한 부분)
[ ] 필요시 계획 수정 요청
```

### 2단계: 완료 기준 협상 (계약)

실행 전 "완료"의 정의를 명확히 한다:

```
각 배치에 대해:
[ ] 달성 목표(deliverables) 정의 -- "이 배치가 끝나면 무엇이 가능한가"
[ ] 검증 방법 명시 -- "어떤 명령어로 완료를 증명하는가"
[ ] 품질 기준 설정 -- "어떤 수준이면 통과인가"
```

계약 형식:
```
배치 N 계약:
- 달성 목표: [구체적 결과물]
- 검증: [명령어 + 예상 결과]
- 품질 기준: [기능성, 코드 품질, 테스트 통과]
```

### 3단계: 배치 실행

```
기본 배치 크기: 3개 작업

실행 규칙:
[ ] 작업 순서 준수 (의존성 고려)
[ ] 각 작업 완료 후 즉시 검증
[ ] 실패 시 즉시 중단 및 보고
```

### 4단계: 독립 평가 (evaluator 에이전트 위임)

각 배치 완료 후 evaluator 에이전트를 별도 컨텍스트로 위임하여 평가:

```
[ ] evaluator 에이전트에 배치 계약 + 변경 범위 전달
[ ] evaluator가 별도 컨텍스트에서 달성 목표 충족 여부 확인
[ ] evaluator가 검증 명령어 실행 (빌드, 테스트, 린트)
[ ] evaluator가 품질 기준별 점수(1-5) 산정
[ ] 임계값(3) 미만 항목에 구체적 수정 지시 포함
```

자기평가 금지: 생성한 당사자가 평가하면 긍정 편향 발생. 반드시 evaluator 에이전트에 위임.

평가 결과:
```
## 배치 N 평가

### 계약 이행
- [x] 목표 1: 충족 (검증: 테스트 통과)
- [ ] 목표 2: 미충족 (원인: ...)

### 품질
| 기준 | 결과 |
|------|------|
| 빌드 | PASS/FAIL |
| 테스트 | PASS/FAIL |
| 기능 동작 | PASS/FAIL |

### 판정
PASS -> 다음 배치 / FAIL -> 수정 후 재평가
```

### 5단계: 피드백 반영 및 반복

FAIL 판정 시:
```
[ ] 피드백의 구체적 수정 사항 확인
[ ] 수정 실행 (최소 변경)
[ ] 재평가 (4단계로 돌아감)
[ ] 최대 3회 반복 -- 3회 실패 시 계획 재검토
```

### 6단계: 완료 및 정리

```
[ ] 모든 배치 완료 확인
[ ] 전체 테스트 실행
[ ] 최종 검증 (빌드, 린트)
[ ] 결과 요약 보고
```

## 배치 실행 규칙

| 규칙 | 설명 |
|------|------|
| 배치 크기 | 기본 3개, 복잡도에 따라 조정 |
| 검증 필수 | 각 작업 후 테스트/빌드 확인 |
| 실패 시 중단 | 다음 작업 진행 전 문제 해결 |
| 보고 주기 | 배치마다 진행 상황 공유 |
| 반복 제한 | 배치당 최대 3회 수정-재평가 |

### 롤백 절차

실패가 복구 불가능할 때:
```
[ ] git stash 또는 git checkout으로 배치 시작 시점 복원
[ ] 실패 원인 분석 후 계획 수정
[ ] 수정된 계획으로 재실행
```

## 실패 처리

작업 실패 시 즉시 중단 -> systematic-debugging 적용 -> 해결 후 재실행. 3회 실패 시 계획 재검토.

## 워크플로우 산출물 (입력)

실행할 계획은 1단계에서 `.claude/workflow/<기능>/plan.md`(또는 사용자가 지정한 계획 문서)를 읽어 온다. 계획 문서가 없으면 먼저 `/plan`으로 생성하거나 사용자에게 확인한 뒤 진행한다. 평가/완료 결과는 같은 폴더에 기록해 다음 세션이 이어받게 한다.

## 에이전트 조율 (옵트인 — 다면적·대규모 작업에 한정)

평상시 단일 작업에는 적용하지 않는다(토큰·지연 비용). 다음에 해당할 때만 전문 에이전트를 팬아웃·조율한다:

- 다중 컴포넌트/계층에 걸친 변경, 대량 마이그레이션, 교차 관점 리뷰(품질+보안+설계)

조율 방식:

1. 의존성 순서로 위임 — explorer(탐색) → architect/planner(설계·계획) → tdd-guide/refactorer(구현) → code-reviewer·security-reviewer(검토) → evaluator(종합 판정). 각 에이전트의 "핸드오프" 정의를 따른다.
2. 독립 단위는 병렬, 의존 단위는 순차. 각 에이전트는 정제된 결과만 반환해 메인 컨텍스트를 보호한다.
3. 생성과 평가를 분리 — 구현 에이전트가 자기 결과를 평가하지 않는다(evaluator에 위임).
4. `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`가 켜져 있으면 활용. 비용·지연이 이득을 넘으면 단일 에이전트로 축소한다.

## 에러 전략 (조율 중 에이전트 실패)

위임한 에이전트가 실패하면 유형별로 대응한다:

- 일시적 오류(네트워크·타임아웃·레이트리밋): 재시도(retry) + 짧은 백오프. 반복되면 다음 전략으로.
- 비핵심·독립 단위: 건너뛰고 계속(skip)하되, 누락을 결과에 반드시 명시한다(무음 금지).
- 핵심 경로: 대체 경로(fallback)로 우회한다(다른 에이전트/접근). 우회 불가하면 즉시 중단.
- 어느 경우든 막히면 전역 사다리(`rules/00-accuracy.md`)를 따른다 — 가용 수단 소진 후 사용자에게 보고.

단일 실행의 기본 실패 처리(중단·최대 3회 재시도·롤백)는 위 "실패 처리"를 따른다.
